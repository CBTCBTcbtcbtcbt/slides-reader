# 后续开发路线

本文档记录第一版之后的开发方向和建议顺序。它不是承诺已经实现的功能，而是后续开发时的上下文和约束。

## 当前 todo

项目根目录 `todo.md` 中记录的方向：

1. 支持 LaTeX 与 Markdown 渲染。
2. 支持 PPT。
3. Prompt engineering。
4. 多模态适配，课程简介结构化。
5. 多样标注。
6. 添加基于讲义 summary 的选择性页面 summary 生成策略。

## 优先级建议

建议按下面顺序推进：

1. 补测试和拆分代码结构。
2. Markdown 与 LaTeX 渲染。
3. 结构化课程简介和 prompt 版本管理。
4. 更好的逐页生成策略。
5. 多样标注。
6. PPT/PPTX 支持。
7. 多用户、任务队列和生产化能力。

这个顺序的原因是：当前代码已经完成主要功能，但后端和前端都集中在单文件中。继续直接叠功能会让维护成本快速升高。

拆分时要保持 API 行为不变。建议先写覆盖主要接口的测试，再迁移代码。

### 前端拆分

优先拆出：

```text
frontend/src/
  api/
    http.ts
    documents.ts
    llm.ts
  types/
    api.ts
  components/
    FilesView/

    ReaderView/
```

拆分目标：

- `App.tsx` 只负责顶层路由和全局状态协调。
- API 调用集中管理。
- 复杂交互逻辑用 hook 封装。
- 样式按组件或模块拆分。

## Markdown 与 LaTeX 渲染

当前 LLM 输出以 Markdown 文本保存，但前端主要按纯文本展示。

建议新增：

- Markdown 渲染库，例如 `react-markdown`。
- 数学公式渲染，例如 `remark-math` 和 `rehype-katex`。
- KaTeX 样式文件。

`LaTeX` 是常用于数学公式排版的语法。`Markdown` 是轻量标记语言，可以表示标题、列表、代码块和链接。

需要注意：

- LLM 输出不应直接作为 HTML 注入页面，避免 XSS 风险。`XSS` 是跨站脚本攻击，指恶意脚本被注入网页执行。
- 如果使用 Markdown 渲染，要明确允许和禁止的 HTML 标签。
- 课程简介、逐页讲稿和问答消息都应该使用同一套安全渲染组件。
- 长公式和长代码块需要移动端适配，不能撑破布局。

建议组件：

```text
MarkdownContent
```

输入：

- `content: string`
- `compact?: boolean`

用于：

- 课程简介。
- 逐页讲稿。
- 讲稿文字块。
- 问答消息。

## PPT/PPTX 支持

当前已经采用第一阶段方案：上传 PPT/PPTX 后，后端通过 LibreOffice 先转换为 PDF，再复用现有 PDF 流程。

已具备：

- 复用现有解析、截图、阅读器和 LLM 流程。
- `documents.file_path` 继续指向系统实际使用的 PDF。
- 前端上传入口支持 PDF/PPT/PPTX。

后续仍可改进：

- 增加转换进度展示。
- 增加更细的转换错误提示。
- 针对字体缺失、动画静态化和复杂版式偏差给出用户提示。

### 路线 B：原生解析 PPTX

后端直接解析 PPTX 文件结构。

优点：

- 可以读取更多 PPT 结构化信息。
- 将来有机会支持备注、版式、对象等信息。

难点：

- 前端展示 PPT 页面更复杂。
- 截图渲染仍然需要转换或专门渲染引擎。

当前暂不长期保存原始 PPT/PPTX，也不新增数据库字段。如果未来需要追踪源文件格式或保留原文件，可以再增加 `source_file_path`、`source_file_type` 等字段。

## Prompt Engineering

`Prompt engineering` 指设计、测试和优化发送给 LLM 的指令文本。

当前 prompt 是普通可编辑文本。后续可以增强：

- prompt 版本号。
- prompt 预设模板。
- 恢复默认 prompt。
- prompt 测试样例。
- 每次生成记录使用的 prompt 快照。

为什么需要 prompt 快照：

- 用户修改 prompt 后，历史讲稿是用旧 prompt 生成的。
- 如果不保存快照，后续无法解释某次输出来自哪个 prompt。

建议新增表：

```sql
CREATE TABLE generation_runs (
  id TEXT PRIMARY KEY,
  document_id TEXT NOT NULL,
  type TEXT NOT NULL,
  prompt_snapshot TEXT NOT NULL,
  model TEXT NOT NULL,
  status TEXT NOT NULL,
  created_at TEXT NOT NULL,
  finished_at TEXT
)
```

第一版不需要立刻实现，但这是后续排查 LLM 输出问题的重要基础。

## 结构化课程简介

当前课程简介是 Markdown 文本。后续可以让 LLM 返回结构化 JSON，再由前端稳定展示。

可能结构：

```json
{
  "topic": "课程主题",
  "audience": "适合对象",
  "key_points": ["知识点 1", "知识点 2"],
  "learning_path": ["步骤 1", "步骤 2"],
  "outcomes": ["学习成果 1", "学习成果 2"]
}
```

优点：

- 前端展示更稳定。
- 可以基于结构做检索和推荐。
- 可以把课程简介作为后续页面生成的更可靠上下文。

风险：

- LLM 可能返回非法 JSON。
- 需要 JSON 修复或重试机制。
- 数据库要兼容旧 Markdown 简介。

建议过渡方案：

- 保留 `course_summary` Markdown 字段。
- 新增 `course_summary_structured` JSON 文本字段。
- 后端同时保存原始输出和解析后的结构化数据。

## 选择性页面 Summary 策略

当前逐页讲稿直接使用课程简介、当前页文字和当前页截图。对于长文档，可能还需要页面级 summary 或章节级 summary。

目标：

- 不必对每一页都做同等成本的深度生成。
- 先识别页面类型和重要性。
- 对重点页生成详细讲稿，对过渡页生成简短讲稿。

可能流程：

1. 上传后解析所有页面。
2. 生成课程简介。
3. 基于课程简介和页面文字，给每页打标签：
   - 封面
   - 目录
   - 概念页
   - 公式页
   - 图表页
   - 例题页
   - 过渡页
   - 总结页
4. 对高价值页面生成详细讲稿。
5. 对低价值页面生成短讲稿或跳过。

需要新增字段：

- `pages.page_type`
- `pages.importance_score`
- `pages.page_summary`
- `pages.generation_strategy`

## 多样标注

当前只有一个讲稿文字块。后续可以支持多种标注：

- 高亮区域。
- 箭头。
- 矩形框。
- 便签。
- 手写笔迹。
- 公式解释卡片。
- 图表注释。

建议新增表：

```sql
CREATE TABLE annotations (
  id TEXT PRIMARY KEY,
  page_id TEXT NOT NULL,
  type TEXT NOT NULL,
  content TEXT,
  geometry_json TEXT NOT NULL,
  style_json TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
)
```

`geometry_json` 保存坐标和形状，`style_json` 保存颜色、线宽等样式。

注意坐标系统：

- 当前讲稿文字块使用阅读区像素坐标。
- 多样标注建议改为相对 PDF 页面坐标，例如 0 到 1 的比例坐标。
- 这样缩放窗口或不同设备打开时，标注仍能对齐页面内容。

## 多模态适配

当前逐页讲稿使用图文输入，课程简介只使用文字输入，问答只使用文字输入。

后续可以增强：

- 课程简介阶段选择关键页面截图加入输入。
- 问答阶段可选加入当前页截图。
- 图表页和公式页优先使用视觉模型。
- 纯文字页使用文本模型降低成本。

需要考虑：

- 不同模型是否支持图片输入。
- 图片大小和 token 成本。
- 失败时是否降级到纯文本。
- 用户在 WebUI 中是否能配置多模态模型。

建议新增配置：

- `text_model`
- `vision_model`
- `enable_vision_for_page_chat`
- `max_images_for_course_summary`

## 后台任务生产化

当前使用 `FastAPI BackgroundTasks`。后续如果任务时间变长、并发变多或需要恢复，建议引入任务队列。

可选方案：

- `Celery + Redis`
- `RQ + Redis`
- `Dramatiq`
- 自己基于 SQLite 的轻量任务表

本地单用户也可以先做 SQLite 任务表：

```sql
CREATE TABLE jobs (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,
  document_id TEXT,
  page_id TEXT,
  status TEXT NOT NULL,
  error_message TEXT,
  created_at TEXT NOT NULL,
  started_at TEXT,
  finished_at TEXT
)
```

优势：

- 进程重启后能看到哪些任务未完成。
- 前端可以显示更准确的任务状态。
- 暂停、继续、取消可以更可靠。

## 多用户和权限

当前没有用户系统。所有文档都在同一个本地数据库中。

如果未来支持多用户，需要：

- `users` 表。
- `documents.user_id`。
- API 鉴权。
- 文件目录按用户隔离。
- 防止用户访问别人的 `document_id`。
- API Key 是全局配置还是用户级配置，需要明确。

在多用户之前，不要把当前接口暴露到公网。

## 安全事项

后续开发要继续保持：

- API Key 不返回明文。
- 文件读取和删除前做路径边界检查。
- 不直接渲染不可信 HTML。
- 上传文件类型后端必须校验，不能只依赖前端。
- 删除文档时先检查路径再删数据库。
- LLM 错误要可见，但不要泄露不必要的敏感信息。
