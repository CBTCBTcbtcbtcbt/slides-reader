# 项目总览

## 产品目标

Slides Reader 是一个围绕 slides 的 AI 陪读和授课工具。它的核心目标不是把 PDF 简单摘要出来，而是让用户在阅读原始课件时，有一个 AI 老师同时完成导读、逐页讲解和当前页答疑。

当前版本围绕本地单用户场景设计。用户上传一份 PDF、PPT 或 PPTX slides 后，系统统一保存或转换为 PDF 并解析页面，再调用 LLM 生成课程简介和逐页讲稿。用户进入阅读器后，可以看到系统实际使用的 PDF 页面、缩略图导航、讲稿文字块、课程简介侧栏，以及围绕当前页的问答区域。系统还支持基于单份课件生成试卷、答题、查看结果、沉淀错题，并基于多份课件生成阶段考试。

## 核心用户流程

1. 用户打开前端页面。
2. 前端请求 `/api/health` 检查后端是否正常。
3. 用户进入设置页配置 LLM 服务地址、API Key、模型名、超时时间和 prompt。
4. 用户选择并上传 PDF/PPT/PPTX slides。
5. 后端把上传文件保存或转换为 `storage/documents/{document_id}.pdf`。
6. 后端创建 `documents` 记录。
7. 后端用 `PyMuPDF` 同步解析每一页文字，并渲染每页 PNG 截图到 `storage/pages/`。
8. PDF 解析成功后，后端把课程简介状态改为 `processing`，并提交后台任务调用 LLM。
9. 课程简介生成成功后，后端把逐页讲稿全部重置为 `pending`，并提交后台任务按页生成讲稿。
10. 前端通过 `/api/documents/{document_id}/status` 轮询生成进度。
11. 用户打开阅读器，前端通过 `/api/documents/{document_id}/file` 加载系统实际使用的 PDF。
12. 前端通过 `/api/documents/{document_id}/pages` 加载页面数据、讲稿文字块和问答历史。
13. 用户拖动或缩放讲稿文字块时，前端通过 `/api/note-blocks/{note_block_id}` 保存位置和尺寸。
14. 用户针对当前页提问时，后端保存用户问题，构造包含当前页上下文的 prompt，调用 LLM 后保存 AI 回答。
15. 用户可以基于单份文档生成试卷，试卷生成完成后进入答题页提交答案。
16. 后端保存答题记录，自动判分，并把答错的题目写入错题本。
17. 用户可以选择多份课件生成阶段考试，阶段考试底层关联一份普通试卷并复用同一答题流程。

## 运行数据与源码边界

项目运行数据默认位于项目根目录的 `storage/`：

```text
storage/
  app.db                    # SQLite 数据库
  documents/                # 系统实际使用的 PDF
  pages/                    # 后端渲染出的每页 PNG 截图
```

这些文件由应用运行时生成，不应作为源码提交。后端可以通过环境变量 `SLIDES_READER_STORAGE_DIR` 指定其他运行数据目录，这对测试很重要，因为测试应该使用临时目录，不能污染开发数据。

## 主要对象

### Document

`Document` 表示用户上传的一份 slides。它对应数据库中的 `documents` 表，也对应前端列表中的一个文档卡片。数据库中的 `file_path` 始终指向系统实际使用的 PDF。

它保存：

- 文档 ID。
- 显示标题。
- 系统实际使用的 PDF 本地路径。
- PDF 解析状态。
- 课程简介正文。
- 课程简介生成状态。
- 逐页讲稿是否暂停。
- 创建时间。

### Page

`Page` 表示某份 PDF 的某一页。它对应数据库中的 `pages` 表。

它保存：

- 页面 ID。
- 所属文档 ID。
- 从 1 开始的页码。
- 解析出的页面文字。
- 后端渲染出的 PNG 截图路径。
- 页面解析状态。
- 页面讲稿正文。
- 页面讲稿生成状态。
- 创建时间。

### NoteBlock

`NoteBlock` 表示阅读器中悬浮在 PDF 周边的讲稿文字块。它对应数据库中的 `note_blocks` 表。

当前每页最多一个 `NoteBlock`。它保存：

- 文字块 ID。
- 所属页面 ID。
- 讲稿正文。
- 左上角坐标 `x` 和 `y`。
- 宽度 `width`。
- 高度 `height`。
- 创建时间和更新时间。

### ChatMessage

`ChatMessage` 表示当前页问答中的一条消息。它对应数据库中的 `chat_messages` 表。

它保存：

- 消息 ID。
- 所属页面 ID。
- 角色，当前只有 `user` 和 `assistant`。
- 消息正文。
- 创建时间。

### AppSetting

`AppSetting` 表示可以在 WebUI 修改的应用配置。它对应数据库中的 `app_settings` 表。

当前主要保存 LLM 配置：

- `base_url`
- `api_key`
- `model`
- `timeout_seconds`
- `course_summary_prompt`
- `lecture_notes_prompt`
- `page_chat_prompt`
- `exam_generation_prompt`

### Exam

`Exam` 表示一份普通试卷。它可以来自单份课件生成，也可以作为阶段考试的底层题目容器。

它保存：

- 试卷 ID。
- 所属主文档 ID。
- 标题和说明。
- 生成状态。
- 总分。
- 创建时间。

### ExamQuestion

`ExamQuestion` 表示试卷中的一道题目。

它保存：

- 题目 ID。
- 所属试卷 ID。
- 题型，当前支持单选题和数字填空题。
- 题干、选项、正确答案和解析。
- 分值、来源页码、难度和知识点标签。

答题页读取题目时会隐藏正确答案和解析。

### ExamAttempt

`ExamAttempt` 表示一次答题记录。

它保存：

- 答题记录 ID。
- 所属试卷 ID。
- 用户答案 JSON。
- 本次得分。
- 创建时间。

### WrongQuestion

`WrongQuestion` 表示错题本中的一条错题记录。

它保存：

- 错题记录 ID。
- 原题目 ID。
- 原试卷 ID。
- 原答题记录 ID。
- 用户当时提交的答案。
- 是否已复习。

### PhaseExam

`PhaseExam` 表示综合多份课件生成的阶段考试。

它保存：

- 阶段考试 ID。
- 阶段考试名称。
- 参与生成的文档 ID 列表。
- 难度。
- 关联的普通试卷 ID。
- 生成状态和失败原因。

## 视图结构

前端当前通过 React Router 管理主要视图：

- 文件页：负责上传、文档列表、生成进度、重命名、删除、讲稿管理、试卷列表和阶段考试列表。
- 阅读页：负责展示系统实际使用的 PDF、缩略图、讲稿文字块、课程简介侧栏和当前页问答。
- 设置页：负责 LLM 配置、prompt 配置和连接测试。
- 答题页：负责展示不含答案的题目并提交答题。
- 结果页：负责展示判分结果，刷新后可以通过 `attempt_id` 恢复。
- 错题本页：负责展示、复习和移除错题。
- 阶段考试创建页：负责选择多份课件并创建综合测试卷。

## 状态流转总览

一份文档从上传到可阅读，通常会经历：

```text
documents.status:
processing -> ready

documents.course_summary_status:
pending -> processing -> ready

pages.lecture_notes_status:
pending -> processing -> ready
```

如果 PDF 损坏或无法解析：

```text
documents.status:
processing -> failed

documents.course_summary_status:
pending -> failed
```

如果课程简介生成失败：

```text
documents.status:
ready

documents.course_summary_status:
processing -> failed
```

如果某页讲稿生成失败：

```text
pages.lecture_notes_status:
processing -> failed
```

这不会影响其他页继续生成或显示已生成结果。

试卷和阶段考试生成通常经历：

```text
exams.status / phase_exams.status:
pending -> processing -> ready
```

如果 LLM 生成、JSON 校验或后台任务失败：

```text
exams.status / phase_exams.status:
processing -> failed
```

阶段考试生成失败时仍会保存关联的 failed `exam_id`，这样删除阶段考试时可以同步清理底层普通试卷。
