# 项目总览

## 产品目标

Slides Reader 是一个围绕 slides 的 AI 陪读和授课工具。它的核心目标不是把 PDF 简单摘要出来，而是让用户在阅读原始课件时，有一个 AI 老师同时完成导读、逐页讲解和当前页答疑。

第一版围绕本地单用户场景设计。用户上传一份 PDF slides 后，系统保存文件并解析页面，再调用 LLM 生成课程简介和逐页讲稿。用户进入阅读器后，可以看到原始 PDF 页面、缩略图导航、讲稿文字块、课程简介侧栏，以及围绕当前页的问答区域。

## 核心用户流程

1. 用户打开前端页面。
2. 前端请求 `/api/health` 检查后端是否正常。
3. 用户进入设置页配置 LLM 服务地址、API Key、模型名、超时时间和 prompt。
4. 用户选择并上传 PDF slides。
5. 后端保存 PDF 到 `storage/documents/`。
6. 后端创建 `documents` 记录。
7. 后端用 `PyMuPDF` 同步解析每一页文字，并渲染每页 PNG 截图到 `storage/pages/`。
8. PDF 解析成功后，后端把课程简介状态改为 `processing`，并提交后台任务调用 LLM。
9. 课程简介生成成功后，后端把逐页讲稿全部重置为 `pending`，并提交后台任务按页生成讲稿。
10. 前端通过 `/api/documents/{document_id}/status` 轮询生成进度。
11. 用户打开阅读器，前端通过 `/api/documents/{document_id}/file` 加载原始 PDF。
12. 前端通过 `/api/documents/{document_id}/pages` 加载页面数据、讲稿文字块和问答历史。
13. 用户拖动或缩放讲稿文字块时，前端通过 `/api/note-blocks/{note_block_id}` 保存位置和尺寸。
14. 用户针对当前页提问时，后端保存用户问题，构造包含当前页上下文的 prompt，调用 LLM 后保存 AI 回答。

- 课程简介生成成功后自动逐页生成讲稿。
- 整份逐页讲稿重新生成。

- 逐页讲稿暂停和继续。

- 底部当前页问答栏折叠。

- 当前页独立问答历史。

## 运行数据与源码边界

项目运行数据默认位于项目根目录的 `storage/`：

```text
storage/
  app.db                    # SQLite 数据库
  documents/                # 用户上传的原始 PDF
  pages/                    # 后端渲染出的每页 PNG 截图
```

这些文件由应用运行时生成，不应作为源码提交。后端可以通过环境变量 `SLIDES_READER_STORAGE_DIR` 指定其他运行数据目录，这对测试很重要，因为测试应该使用临时目录，不能污染开发数据。

## 主要对象

### Document

`Document` 表示用户上传的一份 PDF slides。它对应数据库中的 `documents` 表，也对应前端列表中的一个文档卡片。

它保存：

- 文档 ID。
- 显示标题。
- 原始 PDF 本地路径。
- PDF 解析状态。
- 课程简介正文。
- 课程简介生成状态。
- 逐页讲稿是否暂停。
- 创建时间。

### Page

`Page` 表示某份 PDF 的某一页。它对应数据库中的 `pages` 表。

它保存：

- 页面 ID。
- 所属文档 ID。
- 从 1 开始的页码。
- 解析出的页面文字。
- 后端渲染出的 PNG 截图路径。
- 页面解析状态。
- 页面讲稿正文。
- 页面讲稿生成状态。
- 创建时间。

### NoteBlock

`NoteBlock` 表示阅读器中悬浮在 PDF 周边的讲稿文字块。它对应数据库中的 `note_blocks` 表。

第一版每页最多一个 `NoteBlock`。它保存：

- 文字块 ID。
- 所属页面 ID。
- 讲稿正文。
- 左上角坐标 `x` 和 `y`。
- 宽度 `width`。
- 高度 `height`。
- 创建时间和更新时间。

### ChatMessage

`ChatMessage` 表示当前页问答中的一条消息。它对应数据库中的 `chat_messages` 表。

它保存：

- 消息 ID。
- 所属页面 ID。
- 角色，当前只有 `user` 和 `assistant`。
- 消息正文。
- 创建时间。

### AppSetting

`AppSetting` 表示可以在 WebUI 修改的应用配置。它对应数据库中的 `app_settings` 表。

当前主要保存 LLM 配置：

- `base_url`
- `api_key`
- `model`
- `timeout_seconds`
- `course_summary_prompt`
- `lecture_notes_prompt`
- `page_chat_prompt`

## 视图结构

前端当前有三个视图，由 `currentView` 状态控制：

- `files`：文件管理页，负责上传、文档列表、生成进度、重命名、删除和讲稿管理。
- `reader`：PDF 阅读页，负责展示原始 PDF、缩略图、讲稿文字块、课程简介侧栏和当前页问答。
- `settings`：设置页，负责 LLM 配置和连接测试。

## 状态流转总览

一份文档从上传到可阅读，通常会经历：

```text
documents.status:
processing -> ready

documents.course_summary_status:
pending -> processing -> ready

pages.lecture_notes_status:
pending -> processing -> ready
```

如果 PDF 损坏或无法解析：

```text
documents.status:
processing -> failed

documents.course_summary_status:
pending -> failed
```

如果课程简介生成失败：

```text
documents.status:
ready

documents.course_summary_status:
processing -> failed
```

如果某页讲稿生成失败：

```text
pages.lecture_notes_status:
processing -> failed
```

这不会影响其他页继续生成或显示已生成结果。
