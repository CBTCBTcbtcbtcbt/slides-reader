# 后端架构

## 总体说明

后端当前集中在 `backend/main.py`。这个文件同时包含：

- FastAPI 应用创建。
- CORS 配置。
- 请求体模型。
- 数据库初始化和查询。
- PDF 文件保存、解析和页面截图渲染。
- LLM 配置读取和保存。
- LLMClient 请求封装。
- 课程简介生成。
- 逐页讲稿生成。
- 当前页问答。
- 文档、页面、文字块和聊天相关 API。

这是一种适合第一版快速开发的单文件结构。后续代码继续增长时，建议按文档最后的路线拆分模块。

## 目录和路径常量

后端使用 `Path(__file__).resolve().parent` 计算 `backend/` 的绝对路径，再推导默认存储目录：

```text
BASE_DIR = backend/
DEFAULT_STORAGE_DIR = 项目根目录/storage
STORAGE_DIR = 环境变量 SLIDES_READER_STORAGE_DIR 或 DEFAULT_STORAGE_DIR
DOCUMENT_STORAGE_DIR = STORAGE_DIR/documents
PAGE_IMAGE_STORAGE_DIR = STORAGE_DIR/pages
DATABASE_PATH = STORAGE_DIR/app.db
```

所有路径使用绝对路径可以避免从不同工作目录启动后端时保存位置变化。

## FastAPI 应用

后端应用对象是：

```python
app = FastAPI(
    title="Slides Reader API",
    description="AI slides 阅读与授课工具的后端 API。",
    version="0.1.0",
)
```

`FastAPI` 会根据路由函数和 Pydantic 请求体自动生成 OpenAPI schema。开发时可以访问：

```text
http://127.0.0.1:8000/docs
```

这个页面是 FastAPI 自动生成的 Swagger UI，可以手动调试接口。

## CORS 配置

后端使用 `CORSMiddleware` 开放本地前端访问。`CORS` 是浏览器的跨域资源共享机制；如果前端和后端不是同一个来源，浏览器默认会限制请求。

开发环境中，Vite 已经通过代理转发 `/api`，但保留 CORS 配置可以方便直接从 `localhost:5173` 调用后端。

## 请求体模型

后端使用 Pydantic 的 `BaseModel` 定义 JSON 请求体。`Pydantic` 是 Python 数据校验库，FastAPI 会用它自动校验请求字段。

当前请求体包括：

- `RenameDocumentRequest`
  - `title: str`
  - 用于文档重命名。
- `LLMConfigUpdateRequest`
  - `base_url: str`
  - `api_key: str | None`
  - `model: str`
  - `timeout_seconds: int`
  - `course_summary_prompt: str | None`
  - `lecture_notes_prompt: str | None`
  - `page_chat_prompt: str | None`
  - 用于保存 LLM 配置。
- `LLMTestRequest`
  - `prompt: str`
  - 用于 LLM 连接测试。
- `PageChatRequest`
  - `question: str`
  - 用于当前页问答。
- `NoteBlockPositionUpdateRequest`
  - `x: float`
  - `y: float`
  - `width: float`
  - `height: float`
  - 用于保存讲稿文字块的位置和尺寸。

## 数据库初始化

后端启动时会调用 `init_database()`。该函数会：

1. 创建 `storage/` 目录。
2. 创建或补齐 `documents` 表。
3. 创建或补齐 `pages` 表。
4. 创建 `note_blocks` 表。
5. 创建 `chat_messages` 表。
6. 创建 `app_settings` 表。

SQLite 不支持所有场景下直接 `ALTER TABLE ADD COLUMN IF NOT EXISTS`，所以代码会先用 `PRAGMA table_info(table_name)` 读取已有列，再决定是否添加新列。这让旧版本数据库可以自动迁移到当前字段结构。

## 数据库连接

`get_database_connection()` 会返回一个 `sqlite3.Connection`，并设置：

```python
connection.row_factory = sqlite3.Row
```

这样查询结果既可以像元组一样按位置读取，也可以像字典一样按字段名读取，例如：

```python
row["title"]
```

当前后端每次操作都打开一个连接，使用 `with` 语句自动关闭。第一版没有连接池。

## 序列化函数

后端不会直接把 SQLite 行对象返回给前端，而是通过序列化函数整理字段名：

- `row_to_document(row)`
  - 把 `documents.id` 返回为 `document_id`。
  - 附带 `page_count`。
  - 把 `lecture_notes_paused` 从 SQLite 整数转换为布尔值。
- `row_to_page(row)`
  - 把 `pages.id` 返回为 `page_id`。
  - 把本地 `image_path` 同时转换为可访问的 `image_url`。
  - 如果查询包含 `note_blocks` 字段，会嵌套返回 `note_block`。
  - 默认带空的 `chat_messages`，之后批量填充。
- `row_to_note_block(row)`
  - 返回 `note_block_id`、`page_id`、`content`、坐标、尺寸和时间。
- `row_to_chat_message(row)`
  - 返回 `chat_message_id`、`page_id`、`role`、`content` 和 `created_at`。

这种做法可以把数据库内部字段名和前端 API 字段名隔离开。

## Slides 上传流程

上传入口是：

```text
POST /api/documents
```

流程：

1. `is_supported_slides_upload(file)` 校验文件名后缀必须是 `.pdf`、`.ppt` 或 `.pptx`。
2. 生成 `document_id = uuid4()`。
3. 如果上传的是 PDF，直接保存为 `storage/documents/{document_id}.pdf`。
4. 如果上传的是 PPT/PPTX，先保存到 `storage/conversion-tmp/`，再通过 LibreOffice 转换为 `storage/documents/{document_id}.pdf`，转换后删除临时原文件。
5. 向 `documents` 表写入记录，初始 `status = processing`，`course_summary_status = pending`。
6. 调用 `parse_pdf_pages(document_id, saved_path)` 同步解析 PDF。
7. 如果解析成功，把课程简介状态改为 `processing`，提交 `generate_course_summary(document_id)` 后台任务。
8. 如果解析失败，把课程简介状态改为 `failed`。
9. 返回文档记录、原始文件名和保存后的 PDF 文件名。

注意：PPT/PPTX 转 PDF 和 PDF 解析都是同步执行的，LLM 生成是后台任务。也就是说，大文件上传响应会等待转换、页面解析和截图渲染完成。

`documents.file_path` 永远指向系统实际使用的 PDF 文件。后续阅读器、页面解析、截图、删除和 LLM 输入都只围绕这个 PDF 工作，不长期保存原始 PPT/PPTX。

## PDF 解析和截图

`parse_pdf_pages(document_id, saved_path)` 使用 `pymupdf.open(saved_path)` 打开 PDF。

对每一页：

1. 使用 `page.get_text().strip()` 提取文字。
2. 使用 `render_page_image(page, document_id, page_number)` 渲染 PNG。
3. 调用 `create_page_record(...)` 写入 `pages` 表。

如果某一页失败：

- 该页仍然会创建页面记录。
- `pages.status = failed`。
- `pages.error_message` 保存失败原因。
- 继续处理其他页。

如果整份 PDF 无法打开：

- `documents.status = failed`。
- `documents.error_message` 保存异常。
- 后端服务不会崩溃。

如果所有页面循环完成：

- `documents.status = ready`。

## 页面截图路径

页面截图保存在：

```text
storage/pages/
```

文件名包含文档 ID 和页码，避免不同文档互相覆盖。前端访问时不直接读本地路径，而是请求：

```text
GET /api/documents/{document_id}/pages/{page_number}/image
```

后端返回 `FileResponse`，媒体类型是 `image/png`。

## 文件安全边界

因为数据库中保存了本地文件路径，读取或删除文件前必须检查路径是否还在允许目录内。

相关函数：

- `resolve_document_file_path(file_path)`
- `ensure_document_file_is_safe(file_path)`
- `resolve_page_image_path(image_path)`
- `ensure_page_image_file_is_safe(image_path)`

这些函数会使用 `Path.resolve()` 和 `Path.relative_to()` 确认文件仍在 `storage/documents/` 或 `storage/pages/` 之内。如果路径越界，后端会拒绝访问。

## LLMClient

`LLMClient` 是后端统一的 LLM 调用入口。后续任何业务逻辑都应该使用它，而不是直接散落编写 HTTP 请求。

它提供：

- `complete_text(prompt: str) -> str`
  - 发送纯文本 prompt。
- `complete_with_image(prompt: str, image_path: Path) -> str`
  - 发送文本 prompt 和一张页面截图。
- `_send_chat_completion(messages: list[dict[str, Any]]) -> str`
  - 内部方法，实际发送 OpenAI-compatible chat completions 请求。

请求地址由 `build_chat_completions_url(base_url)` 生成：

- 如果 `base_url` 已经以 `/chat/completions` 结尾，就直接使用。
- 否则拼接 `{base_url}/chat/completions`。

图片会通过 `encode_image_as_data_url(image_path)` 转为 `data:image/png;base64,...` 格式。

## LLM 错误处理

`LLMClient` 会把常见错误整理成明确的 HTTP 错误：

- 配置缺失：`400`
- LLM 服务返回 HTTP 错误：`502`
- 网络连接失败：`502`
- 请求超时：`504`
- 返回 JSON 格式不符合 OpenAI-compatible 结构：`502`
- 空回答：`502`

这让前端可以显示清晰错误，而不是只显示请求失败。

## 后台生成任务

后端使用 FastAPI 的 `BackgroundTasks` 提交生成任务。`BackgroundTasks` 会在响应返回后继续在同一个服务进程里执行函数。

当前后台任务包括：

- `generate_course_summary(document_id)`
- `generate_document_lecture_notes(document_id)`
- `generate_single_page_lecture_notes(document, page)`

重要限制：

- 这是本地开发友好的轻量方案。
- 如果进程退出，后台任务会中断。
- 如果未来需要多用户、任务恢复、任务队列或并发控制，应改用专门的任务系统。

## 逐页讲稿生成锁

后端定义：

```python
LECTURE_NOTES_GENERATION_LOCKS: dict[str, threading.Lock] = {}
LECTURE_NOTES_GENERATION_LOCKS_GUARD = threading.Lock()
```

每个文档有一个生成锁，避免同一进程内对同一文档同时启动多条整份讲稿生成任务。

这个锁只在单进程内有效。如果将来用多个 worker 进程部署，必须改成数据库锁、任务队列锁或其他跨进程机制。

## 讲稿文字块同步

逐页讲稿生成成功后，后端会调用 `sync_note_block_content_for_page(...)`。

规则：

- 如果该页没有 `note_blocks` 记录，就创建默认文字块。
- 如果已经有文字块，只更新 `content` 和 `updated_at`，不覆盖用户拖动过的位置和尺寸。

读取页面列表时，`list_document_pages()` 还会对历史数据做兼容：

- 如果某页讲稿已经是 `ready`，但缺少 `note_blocks`，会补一个默认文字块。

## 当前后端拆分建议

后续如果拆分 `backend/main.py`，建议按下面顺序：

```text
backend/
  app.py                    # FastAPI app 创建和路由挂载
  config.py                 # 路径、默认配置、常量
  database.py               # init_database、get_database_connection
  models.py                 # Pydantic 请求体和 dataclass
  repositories.py           # documents/pages/note_blocks/chat_messages 数据访问
  pdf_service.py            # PDF 解析和截图
  llm_client.py             # LLMConfig、LLMClient
  generation_service.py     # 课程简介和逐页讲稿生成流程
  routes/
    documents.py
    pages.py
    llm.py
    health.py
```

拆分时要先补测试，再移动代码。不要在没有测试保护时同时重构结构和改业务行为。
