# 数据模型与状态机

## SQLite 数据库

项目使用 `SQLite` 保存结构化数据。SQLite 是本地文件数据库，当前数据库默认位于：

```text
storage/app.db
```

后端启动时自动创建和迁移表结构。迁移方式是“启动时检查字段，不存在就补充字段”。当前项目没有引入独立数据库迁移工具。

## documents 表

`documents` 保存每份上传 slides 的文档级信息。无论用户上传 PDF、PPT 还是 PPTX，`file_path` 都保存系统实际使用的 PDF 路径。

字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 文档 ID，使用 `uuid4` 生成。 |
| `title` | `TEXT NOT NULL` | 文档显示标题，初始为用户上传的文件名。 |
| `file_path` | `TEXT NOT NULL` | 系统实际使用的 PDF 在本地的绝对路径。 |
| `status` | `TEXT NOT NULL` | PDF 解析状态。 |
| `error_message` | `TEXT` | PDF 解析或文档级错误原因。 |
| `course_summary` | `TEXT` | LLM 生成的课程简介正文。 |
| `course_summary_status` | `TEXT NOT NULL DEFAULT 'pending'` | 课程简介生成状态。 |
| `course_summary_error` | `TEXT` | 课程简介生成失败原因。 |
| `lecture_notes_paused` | `INTEGER NOT NULL DEFAULT 0` | 是否暂停逐页讲稿生成，`0` 表示未暂停，`1` 表示暂停。 |
| `created_at` | `TEXT NOT NULL` | UTC ISO 格式创建时间。 |

API 返回时，`id` 会序列化为 `document_id`，`lecture_notes_paused` 会转为布尔值。

## pages 表

`pages` 保存 PDF 每一页的解析结果和讲稿状态。

字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 页面 ID，使用 `uuid4` 生成。 |
| `document_id` | `TEXT NOT NULL` | 所属文档 ID。 |
| `page_number` | `INTEGER NOT NULL` | 从 1 开始的页码。 |
| `text` | `TEXT NOT NULL` | 从 PDF 页面提取出的文字，空白页可以是空字符串。 |
| `image_path` | `TEXT` | 页面 PNG 截图本地路径。 |
| `status` | `TEXT NOT NULL` | 页面解析状态。 |
| `error_message` | `TEXT` | 页面解析或截图渲染错误原因。 |
| `lecture_notes` | `TEXT` | LLM 生成的当前页讲稿。 |
| `lecture_notes_status` | `TEXT NOT NULL DEFAULT 'pending'` | 当前页讲稿生成状态。 |
| `lecture_notes_error` | `TEXT` | 当前页讲稿生成失败原因。 |
| `created_at` | `TEXT NOT NULL` | UTC ISO 格式创建时间。 |

约束：

```sql
UNIQUE(document_id, page_number)
FOREIGN KEY(document_id) REFERENCES documents(id)
```

API 返回时，`id` 会序列化为 `page_id`，并额外返回 `image_url`。

## note_blocks 表

`note_blocks` 保存阅读器里的讲稿文字块。

字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 文字块 ID。 |
| `page_id` | `TEXT NOT NULL` | 所属页面 ID。 |
| `content` | `TEXT NOT NULL` | 文字块正文，来自当前页讲稿。 |
| `x` | `REAL NOT NULL` | 左上角 x 坐标，单位是像素。 |
| `y` | `REAL NOT NULL` | 左上角 y 坐标，单位是像素。 |
| `width` | `REAL NOT NULL` | 宽度，单位是像素。 |
| `height` | `REAL NOT NULL` | 高度，单位是像素。 |
| `created_at` | `TEXT NOT NULL` | 创建时间。 |
| `updated_at` | `TEXT NOT NULL` | 更新时间。 |

约束：

```sql
UNIQUE(page_id)
FOREIGN KEY(page_id) REFERENCES pages(id)
```

当前实现每页只有一个讲稿文字块。

默认尺寸和位置：

| 常量 | 当前值 | 说明 |
| --- | --- | --- |
| `DEFAULT_NOTE_BLOCK_X` | `24.0` | 默认 x 坐标。 |
| `DEFAULT_NOTE_BLOCK_Y` | `24.0` | 默认 y 坐标。 |
| `DEFAULT_NOTE_BLOCK_WIDTH` | `320.0` | 默认宽度。 |
| `DEFAULT_NOTE_BLOCK_HEIGHT` | `240.0` | 默认高度。 |
| `MIN_NOTE_BLOCK_WIDTH` | `120.0` | 后端允许保存的最小宽度。 |
| `MIN_NOTE_BLOCK_HEIGHT` | `80.0` | 后端允许保存的最小高度。 |

## chat_messages 表

`chat_messages` 保存每一页自己的问答历史。

字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 消息 ID。 |
| `page_id` | `TEXT NOT NULL` | 所属页面 ID。 |
| `role` | `TEXT NOT NULL` | 消息角色，只允许业务层写入 `user` 或 `assistant`。 |
| `content` | `TEXT NOT NULL` | 消息正文。 |
| `created_at` | `TEXT NOT NULL` | 创建时间。 |

约束：

```sql
FOREIGN KEY(page_id) REFERENCES pages(id)
```

问答历史按 `created_at ASC` 返回。

## chat_attachments 表

`chat_attachments` 保存当前页问答消息关联的图片附件。

字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | 附件 ID。 |
| `chat_message_id` | `TEXT NOT NULL` | 所属聊天消息 ID。 |
| `page_id` | `TEXT NOT NULL` | 所属页面 ID，便于按页读取最近历史图片。 |
| `kind` | `TEXT NOT NULL` | 附件类型，当前固定为 `image`。 |
| `original_filename` | `TEXT NOT NULL` | 用户上传或粘贴图片时的文件名。 |
| `mime_type` | `TEXT NOT NULL` | 后端根据文件头识别出的真实图片类型。 |
| `file_path` | `TEXT NOT NULL` | 图片保存在 `storage/chat-attachments/` 下的本地路径。 |
| `file_size` | `INTEGER NOT NULL` | 图片字节数。 |
| `display_order` | `INTEGER NOT NULL` | 同一条消息内的附件展示顺序。 |
| `created_at` | `TEXT NOT NULL` | 创建时间。 |

约束：

```sql
FOREIGN KEY(chat_message_id) REFERENCES chat_messages(id)
FOREIGN KEY(page_id) REFERENCES pages(id)
```

当前支持：

- 图片类型：PNG、JPEG、WebP。
- 单轮最多 4 张图片。
- 单张图片最大 10MB。
- 后端按文件头识别图片类型，不只依赖前端 MIME。

## app_settings 表

`app_settings` 保存可在 WebUI 修改的配置。

字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `key` | `TEXT PRIMARY KEY` | 配置名。 |
| `value` | `TEXT NOT NULL` | 配置值。 |
| `updated_at` | `TEXT NOT NULL` | 更新时间。 |

写入时使用：

```sql
INSERT INTO app_settings (key, value, updated_at)
VALUES (?, ?, ?)
ON CONFLICT(key) DO UPDATE SET
  value = excluded.value,
  updated_at = excluded.updated_at
```

也就是说，配置 key 已存在时会覆盖旧值。

## Document 解析状态

`documents.status` 表示系统实际使用的 PDF 文件解析层面的状态。

| 状态 | 含义 | 典型来源 |
| --- | --- | --- |
| `processing` | 已保存或转换为 PDF，正在解析页面和截图。 | 上传后立即写入。 |
| `ready` | PDF 页面解析和截图流程完成。 | `parse_pdf_pages` 正常结束。 |
| `failed` | 整份 PDF 无法打开或整体解析失败。 | PDF 损坏、无法读取。 |
| `uploaded` | 保留的兼容状态，当前主流程基本不用。 | 兼容历史数据。 |

当前上传流程中，文档从 `processing` 开始，解析成功变为 `ready`，解析失败变为 `failed`。

## Page 解析状态

`pages.status` 表示单页解析和截图状态。

| 状态 | 含义 |
| --- | --- |
| `ready` | 该页文字提取和截图渲染成功。 |
| `failed` | 该页处理失败，但页面记录仍然存在。 |

空白页也应该创建记录。空白页的 `text` 可以为空字符串，但不应该被跳过。

## Course Summary 状态

`documents.course_summary_status` 表示课程简介生成状态。

| 状态 | 含义 |
| --- | --- |
| `pending` | 等待生成或尚未开始。 |
| `processing` | 正在后台调用 LLM。 |
| `ready` | 已成功生成，`course_summary` 应有正文。 |
| `failed` | 生成失败，`course_summary_error` 应有原因。 |

课程简介只在 `documents.status = ready` 后才应该开始生成。

## Lecture Notes 状态

`pages.lecture_notes_status` 表示某一页讲稿生成状态。

| 状态 | 含义 |
| --- | --- |
| `pending` | 等待生成。 |
| `processing` | 正在生成。 |
| `ready` | 已生成成功，`lecture_notes` 应有正文。 |
| `failed` | 生成失败，`lecture_notes_error` 应有原因。 |

逐页讲稿依赖课程简介。只有 `course_summary_status = ready` 且 `course_summary` 非空时，后端才允许生成讲稿。

## Exam 状态

`exams.status` 和 `phase_exams.status` 使用相同的生成状态语义。

| 状态 | 含义 |
| --- | --- |
| `pending` | 记录已创建，后台生成任务尚未正式开始。 |
| `processing` | 后台正在生成题目。 |
| `ready` | 题目已生成成功，可以答题。 |
| `failed` | 生成失败，`error_message` 应有原因。 |

阶段考试会关联一份普通试卷：

```text
phase_exams.exam_id -> exams.id
```

答题、判分、错题记录都复用普通试卷链路。

## 主要状态流转

### 正常上传和生成

```text
POST /api/documents

documents.status = processing
documents.course_summary_status = pending
pages.lecture_notes_status = pending

parse_pdf_pages 成功

documents.status = ready
documents.course_summary_status = processing

generate_course_summary 成功

documents.course_summary_status = ready
pages.lecture_notes_status = pending

generate_document_lecture_notes 按页执行

page 1 lecture_notes_status = processing -> ready
page 2 lecture_notes_status = processing -> ready
...
```

### PDF 解析失败

```text
documents.status = processing

parse_pdf_pages 抛出异常

documents.status = failed
documents.error_message = 异常原因
documents.course_summary_status = failed
documents.course_summary_error = "PDF 解析失败，无法生成课程简介。"
```

### 课程简介失败

```text
documents.status = ready
documents.course_summary_status = processing

LLM 调用失败

documents.course_summary_status = failed
documents.course_summary_error = 错误原因
pages.lecture_notes_status 保持原状
```

### 单页讲稿失败

```text
pages.lecture_notes_status = processing

页面解析失败或 LLM 调用失败

pages.lecture_notes_status = failed
pages.lecture_notes_error = 错误原因

其他页面继续处理
```

### 暂停和继续

暂停接口：

```text
POST /api/documents/{document_id}/lecture-notes/pause
```

效果：

```text
documents.lecture_notes_paused = 1
```

暂停不会中断已经发给 LLM 的当前页请求，只会阻止后台生成任务开始下一页。

继续接口：

```text
POST /api/documents/{document_id}/lecture-notes/resume
```

效果：

```text
documents.lecture_notes_paused = 0
```

然后后台任务会继续生成尚未 `ready` 的页面。继续生成时，后端不会覆盖已 `ready` 的讲稿。

## 状态接口聚合规则

`GET /api/documents/{document_id}/status` 会聚合：

- 文档状态。
- 课程简介状态。
- 页面总数。
- 讲稿 `ready` 页数。
- 讲稿 `failed` 页数。
- 讲稿 `processing` 页数。
- 讲稿 `pending` 页数。
- 是否暂停。
- 是否建议前端继续轮询。
- 每页简要状态。

`should_poll` 为 `true` 的条件是：

- 文档仍在 `processing`。
- 或课程简介仍在 `processing`。
- 或未暂停，且有讲稿页面正在 `processing`。

前端只要看到 `should_poll = false`，就可以停止该文档的状态轮询。

## 考试数据清理规则

删除试卷时，后端按下面顺序清理：

```text
wrong_questions -> exam_attempts -> exam_questions -> exams
```

删除阶段考试时，后端会先读取 `phase_exams.exam_id`，如果存在关联普通试卷，也按上面的试卷删除顺序清理，再删除 `phase_exams` 记录。

删除文档时，后端会删除：

- 文档直接关联的普通试卷。
- `document_ids` 中包含该文档的阶段考试记录。
- 上述试卷关联的题目、答题记录和错题。

这些清理规则是为了避免出现孤儿试卷、孤儿题目或无法从 UI 清理的错题记录。
