# 任务 05：渲染页面截图

## 1. 目标

本任务的目标是为 PDF 的每一页生成一张图片。

这些页面截图会用于两个地方：

- 前端可以在需要时显示页面图片。
- LLM 可以结合页面截图理解 slides 中的图表、流程图、公式和排版。

## 2. 实现内容

- 使用 `PyMuPDF` 将每一页 PDF 渲染为 PNG 图片。
  - PNG 是常见图片格式，适合保存清晰的页面截图。
- 将页面图片保存到本地目录，例如 `storage/pages/`。
- 在 `Page` 数据表中增加或使用 `image_path` 字段。
- 每一页的 `image_path` 保存对应 PNG 图片路径。
- 提供后端接口，使前端可以访问页面图片。
- 确保图片页码和 PDF 页码一一对应。

## 3. 实现途径 / 方法 / 手段

- 在 PDF 文字解析流程之后，继续对每页进行渲染。
- 使用 `PyMuPDF` 打开 PDF 文件并读取页面对象。
- 对每一页调用渲染方法，把页面转换为像素图。
  - 像素图可以理解为图片在内存中的表示。
- 保存图片时，建议文件名包含 `document_id` 和页码。
  - 这样可以避免不同文档的页面图片重名。
- 渲染分辨率需要足够清晰。
  - 分辨率过低会影响前端展示和 LLM 视觉理解。
  - 分辨率过高会增加存储空间和 LLM 调用成本。
- 推荐第一版使用中等清晰度，优先保证稳定可用。
- 图片访问接口可以直接返回静态文件。
  - 静态文件指不需要动态计算、可以直接读取返回的文件，例如图片、PDF。
- 如果某一页图片生成失败，该页状态应标记为失败，但不应影响其他页面继续处理。

## 4. 合格标准

- 上传 PDF 并完成处理后，每一页都有对应 PNG 图片。
- 本地 `storage/pages/` 目录中可以看到生成的页面图片。
- 图片打开后内容与 PDF 对应页一致。
- 第一页图片对应第一页，第二页图片对应第二页，不能错位。
- 后端返回页面详情时包含图片访问信息。
- 前端或浏览器可以通过接口打开页面图片。
- 当前任务不要求调用 LLM。

## 5. 不合格情况

- 页面图片没有生成。
- 图片为空白、严重模糊或内容被裁剪。
- 图片页码和 PDF 页码对应错误。
- 图片路径没有保存到数据库。
- 某一页生成失败导致全部页面都不可用。
- 当前任务加入复杂前端阅读器或 LLM 讲稿功能，导致截图生成无法单独验证。
