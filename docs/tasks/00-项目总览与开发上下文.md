# 00：项目总览与开发上下文

## 1. 项目一句话概述

本项目是一个使用 Python 后端和 WebUI 前端实现的 AI slides 阅读与授课工具。

用户上传 PDF 格式的 slides 后，系统会调用 LLM 阅读整份文档，并让 LLM 扮演老师：先生成整门课的课程简介，再逐页生成讲课重点。用户在网页中看到的仍然是原始 slides，但每一页周边会显示可拖动的 AI 老师讲课文字块，并且用户可以针对当前页单独向 AI 老师提问。

这里的 LLM 是 Large Language Model 的缩写，中文通常叫“大语言模型”。在本项目中，LLM 不是简单做摘要，而是要模拟老师讲课、解释知识点和回答学生问题。

## 2. 项目目标

第一版项目要完成一个本地可用的学习闭环：

- 用户可以上传 PDF slides。
- 后端可以解析 PDF 的页数、文字和页面截图。
- LLM 可以先根据整份 slides 生成课程简介。
- LLM 可以再根据每一页内容生成当前页讲稿。
- 前端可以展示原始 PDF 页面。
- 前端可以显示当前页的 AI 老师讲稿文字块。
- 讲稿文字块可以拖动，并且位置可以保存。
- 用户可以用按钮和鼠标中键翻页。
- 用户可以在每页底部向 AI 老师提问。
- 每页问答历史独立保存。

第一版优先保证功能闭环和代码清晰，不追求复杂账号系统、多人协作、云端部署或高级富文本编辑能力。

## 3. 用户使用流程

用户的预期使用流程如下：

1. 打开 WebUI。
2. 上传一份 PDF slides。
3. 系统保存 PDF，并开始后台处理。
4. 系统解析 PDF，提取每页文字，并渲染每页截图。
5. 系统把整份文档内容发送给 LLM，生成课程简介。
6. 系统逐页调用 LLM，生成每一页的老师讲稿。
7. 用户进入阅读界面，先查看课程简介。
8. 用户逐页阅读原始 slides。
9. 用户查看每页周边的 AI 老师讲课文字块。
10. 用户可以拖动文字块，避免遮挡 slides 内容。
11. 用户可以通过按钮或鼠标中键翻页。
12. 用户可以在当前页下方输入问题。
13. LLM 根据当前页内容、当前页讲稿和当前页问答历史回答问题。

## 4. 技术路线

后端使用 `Python + FastAPI`。

- `FastAPI` 是 Python Web 框架，用来编写 HTTP API。
- HTTP API 可以理解为前端和后端之间约定好的访问入口。
- 后端负责上传文件、保存数据、解析 PDF、调用 LLM、返回页面数据。

前端使用 `React + TypeScript + Vite`。

- `React` 是构建网页界面的 JavaScript 库。
- `TypeScript` 是带类型检查的 JavaScript，可以减少前端接口和变量使用错误。
- `Vite` 是前端开发工具，用来启动开发服务器和打包前端代码。
- 前端负责 PDF 展示、翻页、拖动文字块、显示进度和提交问题。

PDF 处理使用 `PyMuPDF`。

- `PyMuPDF` 是 Python 中常用的 PDF 处理库。
- 它可以读取 PDF 页数、提取每页文字、把每页渲染成图片。

数据保存使用 `SQLite + 本地文件夹`。

- `SQLite` 是轻量级本地数据库，适合第一版本地单用户应用。
- 上传的 PDF 和页面截图保存在本地文件夹中。
- 文档记录、页面记录、讲稿、文字块位置和问答历史保存在 SQLite 中。

LLM 调用使用 `OpenAI-compatible API`。

- `OpenAI-compatible API` 指使用类似 OpenAI 的接口格式。
- 这样后端既可以连接 OpenAI，也可以连接第三方兼容服务。
- 配置项至少包括 `LLM_BASE_URL`、`LLM_API_KEY` 和 `LLM_MODEL`。

## 5. 核心数据理解

后续开发时，应优先围绕以下几个核心对象理解项目。

### Document

`Document` 表示用户上传的一份 PDF slides。

它应该保存：

- 文档 ID。
- 文档标题或原始文件名。
- PDF 文件路径。
- 文档处理状态。
- 课程简介。
- 创建时间。

文档处理状态可以包括：

- `uploaded`：文件已上传。
- `processing`：正在解析或生成 AI 内容。
- `ready`：主要内容已经处理完成。
- `failed`：处理失败。

### Page

`Page` 表示 PDF 中的某一页。

它应该保存：

- 页面 ID。
- 所属文档 ID。
- 页码。
- 页面文字。
- 页面截图路径。
- 当前页讲稿。
- 页面处理状态。
- 失败错误信息。

页面是课程讲稿和页面问答的核心单位。

### NoteBlock

`NoteBlock` 表示前端中可拖动的讲课文字块。

它应该保存：

- 所属页面 ID。
- 文字块内容。
- 横向位置 `x`。
- 纵向位置 `y`。
- 宽度 `width`。
- 高度 `height`。

第一版每页默认一个讲稿文字块即可，不需要支持用户创建多个自定义文字块。

### ChatMessage

`ChatMessage` 表示某一页中的一条问答消息。

它应该保存：

- 所属页面 ID。
- 消息角色。
- 消息内容。
- 创建时间。

消息角色可以是：

- `user`：用户提出的问题。
- `assistant`：LLM 老师的回答。

每页问答历史必须独立保存，不能所有页面共用一份聊天记录。

## 6. LLM 工作流

LLM 相关逻辑需要分成三个阶段。

### 阶段一：生成课程简介

输入内容：

- 文档标题。
- 总页数。
- 每页提取出的文字。

输出内容：

- 课程主题。
- 学习对象。
- 主要知识点。
- 推荐学习顺序。
- 学完后应掌握的内容。

输出保存位置：

- `Document.course_summary`。

### 阶段二：逐页生成讲稿

每一页单独调用一次 LLM。

输入内容：

- 课程简介。
- 当前页页码。
- 当前页文字。
- 当前页截图。

输出内容：

- 当前页核心观点。
- 老师讲解词。
- 学生需要注意的地方。
- 与前后页面的衔接。

输出保存位置：

- `Page.lecture_notes`。

### 阶段三：当前页问答

用户在某一页提问时调用 LLM。

输入内容：

- 课程简介。
- 当前页文字。
- 当前页讲稿。
- 当前页历史问答。
- 用户最新问题。

输出内容：

- 面向学生的老师式回答。

输出保存位置：

- `ChatMessage` 中的 `assistant` 消息。

## 7. WebUI 总体理解

WebUI 不应该做成普通文件管理页面，而应该做成学习工具。

阅读页是核心页面，它应该包含：

- 文档标题和处理状态。
- 课程简介入口。
- PDF 当前页展示区。
- 明显的上一页和下一页按钮。
- 鼠标中键翻页能力。
- 当前页 AI 老师讲稿文字块。
- 可拖动并可保存位置的文字块。
- 底部当前页问答框。
- 当前页问答历史。

用户看到的主体仍然应该是原始 slides，而不是把 slides 转换成纯文本文章。

## 8. 推荐 API 轮廓

后端 API 可以按以下方向设计：

- `GET /api/health`
  - 检查后端是否正常运行。
- `POST /api/documents`
  - 上传 PDF，并创建文档记录。
- `GET /api/documents`
  - 获取已上传文档列表。
- `GET /api/documents/{document_id}`
  - 获取文档详情。
- `GET /api/documents/{document_id}/file`
  - 返回原始 PDF 文件。
- `GET /api/documents/{document_id}/status`
  - 获取文档和页面处理进度。
- `GET /api/documents/{document_id}/pages/{page_number}`
  - 获取某一页的文字、截图、讲稿、文字块和问答历史。
- `PATCH /api/note-blocks/{note_block_id}`
  - 保存文字块位置。
- `POST /api/pages/{page_id}/chat`
  - 针对当前页提问。
- `POST /api/pages/{page_id}/regenerate`
  - 重新生成某一页讲稿。

具体实现时可以根据代码结构微调路径，但不要改变核心职责。

## 9. 任务拆分顺序

当前项目已经被拆成 14 个开发任务。后续开发应按顺序推进，不建议跳跃实现。

1. `01-创建项目骨架.md`
   - 建立最小可运行的前后端。
2. `02-实现PDF上传.md`
   - 实现 PDF 文件上传和本地保存。
3. `03-加入SQLite数据库.md`
   - 保存文档记录，支持重启后数据不丢失。
4. `04-解析PDF页数和文字.md`
   - 提取每页文字，创建页面记录。
5. `05-渲染页面截图.md`
   - 为每页生成 PNG 截图。
6. `06-实现LLM配置和客户端.md`
   - 封装统一 LLM 调用入口。
7. `07-生成课程简介.md`
   - 根据整份文档生成课程简介。
8. `08-逐页生成讲稿.md`
   - 为每页生成老师讲稿。
9. `09-实现PDF阅读界面.md`
   - 在 WebUI 中展示原始 PDF 并支持翻页。
10. `10-展示可拖动讲课文字块.md`
    - 展示讲稿文字块并保存拖动位置。
11. `11-实现当前页问答.md`
    - 针对当前页提问，并保存每页独立历史。
12. `12-处理生成进度和错误状态.md`
    - 展示处理进度、错误和重试能力。
13. `13-整理界面布局.md`
    - 整合阅读、讲稿、问答和简介，形成可用学习界面。
14. `14-补充测试.md`
    - 补充后端、前端和手动验收测试。

每个任务都应该只添加一个主要功能，并且必须能单独验证合格或不合格。

## 10. 开发约束

后续开发必须遵守以下约束：

- 所有对话、代码注释和技术文档使用中文。
- 重要英文技术名词保留英文原文，并在第一次出现时解释。
- 用户默认只了解 Python 基础语法，不默认了解 Web、数据库、前端框架或 LLM API。
- 代码注释要详细，尤其是关键业务逻辑部分。
- 不要把 API Key 写死在代码中。
- 不要把上传的 PDF、页面截图和数据库文件当作源码提交。
- 第一版按本地单用户应用设计，不做登录和权限系统。
- 第一版优先完成闭环，不提前引入复杂微服务、消息队列或云部署。
- LLM 输出先按普通 Markdown 文本保存，不做复杂富文本编辑器。

## 11. 第一版不做的内容

为了控制范围，第一版暂不实现以下能力：

- 用户注册和登录。
- 多用户权限管理。
- 在线协作。
- 云存储。
- 复杂后台任务队列。
- 多个自定义讲稿文字块。
- 富文本讲稿编辑器。
- 跨文档知识库问答。
- 自动生成语音讲课。
- 自动生成视频课程。

这些功能可以在第一版核心流程稳定后再考虑。

## 12. 给后续开发模型的注意事项

后续开发模型在接手任务时，应先阅读本文件，再阅读当前要执行的具体任务文档。

执行某个任务时要注意：

- 不要一次实现多个任务的功能。
- 不要为了“顺手”提前做后续任务。
- 每个任务完成后，都要用任务文档中的合格标准验证。
- 如果需要引入新的依赖，要确认它确实服务于当前任务。
- 如果修改数据模型，要考虑后续任务是否依赖这个字段。
- 如果 LLM 调用暂时无法真实测试，应使用 mock 或清晰的错误处理保证其他流程可测试。
- 如果前端功能依赖后端数据，应先保证后端接口返回结构稳定。
- 如果发现任务文档与当前代码冲突，应优先保持项目可运行，并更新文档说明原因。

本项目的核心不是“把 PDF 摘要出来”，而是“围绕 slides 建立一个 AI 老师陪读和讲课的学习体验”。后续实现应始终围绕这个目标取舍功能。
