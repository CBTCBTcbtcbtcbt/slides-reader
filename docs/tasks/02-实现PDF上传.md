# 任务 02：实现 PDF 上传

## 1. 目标

本任务的目标是让用户可以在网页中上传 PDF 格式的 slides 文件，并让后端把上传的文件保存到本地。

完成后，项目应该具备最基础的文档输入能力。后续的 PDF 解析、页面截图、LLM 讲解都依赖这个上传结果。

## 2. 实现内容

- 前端添加一个 PDF 上传区域。
- 上传区域可以选择本地 `.pdf` 文件。
- 前端将选择的文件发送给后端。
- 后端新增 `POST /api/documents` 接口。
- 后端检查上传文件是否为 PDF。
- 后端把合法 PDF 保存到本地目录，例如 `storage/documents/`。
- 后端返回一个 `document_id`。
  - `document_id` 是文档的唯一标识，后续查询、解析、阅读都会通过它找到对应文档。
- 上传失败时，前端显示清楚的错误信息。

## 3. 实现途径 / 方法 / 手段

- 前端使用 HTML 文件选择控件。
  - 文件选择控件是浏览器原生能力，可以让用户从本地选择文件。
- 前端使用 `FormData` 上传文件。
  - `FormData` 是浏览器提供的对象，适合提交文件和普通字段。
- 后端使用 FastAPI 的文件上传能力。
  - FastAPI 可以通过上传文件类型接收前端发来的文件。
- 后端校验至少包含两层：
  - 文件名后缀必须是 `.pdf`。
  - 文件的 `content-type` 应该符合 PDF 类型。
- 保存文件时，需要避免直接使用用户上传的原始文件名作为最终文件名。
  - 原始文件名可能重复，也可能包含不适合做路径的字符。
  - 推荐使用后端生成的唯一 ID 作为文件名的一部分。
- 本任务可以暂时不写入数据库。
  - 只要后端能保存文件并返回 `document_id` 即可。
  - 数据库持久化会在任务 03 中完成。
- 前端上传成功后，页面显示上传成功和返回的 `document_id`。

## 4. 合格标准

- 用户可以在前端选择一个 PDF 文件并提交。
- 后端成功接收文件后，返回 HTTP 200 或 HTTP 201。
- 后端响应中包含 `document_id`。
- 上传成功后，本地 `storage/documents/` 目录中可以看到保存的 PDF 文件。
- 上传非 PDF 文件时，后端拒绝请求。
- 上传非 PDF 文件时，前端能显示明确错误，而不是页面无响应。
- 当前任务只验证文件上传和保存，不要求解析 PDF 内容。

## 5. 不合格情况

- 前端没有上传入口。
- 后端没有 `POST /api/documents` 接口。
- PDF 文件发送到后端后没有保存到本地。
- 上传非 PDF 文件也能成功。
- 后端返回结果中没有可用于后续操作的 `document_id`。
- 上传失败时用户看不到错误原因。
- 当前任务直接加入复杂数据库逻辑或 LLM 调用，导致上传功能无法单独验证。
