# 任务 06：实现 LLM 配置和客户端

## 1. 目标

本任务的目标是为后端建立一个统一的 LLM 调用入口。

LLM 是 Large Language Model 的缩写，中文通常叫“大语言模型”。本项目中，LLM 会扮演老师，负责生成课程简介、逐页讲稿和回答问题。

完成后，后端应该能通过 OpenAI-compatible API 调用模型，并且可以通过配置切换不同服务商。

## 2. 实现内容

- 增加 LLM 相关配置项。
- 配置项至少包括：
  - `LLM_BASE_URL`：模型服务地址。
  - `LLM_API_KEY`：模型服务密钥。
  - `LLM_MODEL`：使用的模型名称。
- 创建统一的 `LLMClient`。
  - `LLMClient` 是封装 LLM 调用逻辑的后端模块。
  - 其他业务代码只调用它，不直接关心底层 API 细节。
- 使用 OpenAI-compatible API 格式。
  - OpenAI-compatible API 指第三方服务提供了和 OpenAI 类似的请求格式。
  - 这样可以用同一套代码接入不同模型服务商。
- 支持普通文本请求。
- 支持带页面截图的视觉请求。
- 处理 LLM 调用失败的情况。

## 3. 实现途径 / 方法 / 手段

- 使用环境变量读取配置。
  - 环境变量是操作系统或运行环境中保存配置的方式。
  - API Key 必须放在环境变量或 `.env` 文件中，不能写死在代码里。
- 可以使用官方 OpenAI Python SDK。
  - SDK 是服务商提供的软件开发工具包，可以简化 API 调用。
  - 即使使用第三方 API，只要兼容 OpenAI 格式，也可以设置自定义 `base_url`。
- `LLMClient` 建议至少提供两个方法：
  - 发送文本消息并返回文本回答。
  - 发送文本加图片并返回文本回答。
- 图片传给 LLM 前，需要转换为模型接口要求的格式。
  - 常见方式是把图片转为 base64。
  - base64 是一种把二进制文件转换为文本的编码方式。
- 所有 LLM 返回结果先按普通 Markdown 文本保存。
  - Markdown 是一种轻量文本格式，可以表示标题、列表、代码块等。
- 调用失败时，需要返回明确错误。
  - 例如缺少 API Key、网络请求失败、模型返回错误。

## 4. 合格标准

- 后端可以从环境变量读取 LLM 配置。
- 代码中没有硬编码 API Key。
- 使用 OpenAI-compatible 服务时，后端能成功发起一次文本请求并得到回答。
- 如果配置错误，后端能给出明确错误信息。
- `LLMClient` 的调用方式统一，后续功能不需要重复写底层请求代码。
- 支持为后续页面截图请求预留图片输入能力。
- 当前任务只验证 LLM 客户端可用，不要求生成课程简介或逐页讲稿。

## 5. 不合格情况

- API Key 写死在代码里。
- 只能连接某一个固定平台，不能配置 `base_url`。
- LLM 请求逻辑散落在多个业务文件中。
- 调用失败时只返回模糊错误，无法判断是配置问题还是网络问题。
- 不支持后续传入页面截图。
- 当前任务直接实现完整课程生成流程，导致 LLM 客户端无法独立测试。
