# 任务 03：加入 SQLite 数据库

## 1. 目标

本任务的目标是让上传的文档记录可以持久保存。

“持久保存”指的是后端重启之后，之前上传过的文档记录仍然存在。这样用户刷新网页或重新启动项目后，还能看到已经上传过的 slides。

## 2. 实现内容

- 在后端项目中加入 `SQLite` 数据库。
  - `SQLite` 是一种轻量级数据库，数据保存在本地单个文件中，适合本地开发和单用户应用。
- 创建 `Document` 数据表。
- 上传 PDF 成功后，向 `Document` 表写入一条记录。
- `Document` 至少保存以下信息：
  - 文档 ID。
  - 文档标题或原始文件名。
  - PDF 文件保存路径。
  - 文档处理状态。
  - 创建时间。
- 新增 `GET /api/documents` 接口，用来返回所有已上传文档。
- 前端显示已上传文档列表。

## 3. 实现途径 / 方法 / 手段

- 可以使用 `SQLModel`、`SQLAlchemy` 或类似 ORM 工具。
  - ORM 是对象关系映射工具，可以让开发者用 Python 类操作数据库表。
  - 如果项目追求简单，`SQLModel` 会比较适合，因为它和 FastAPI 的类型风格接近。
- 数据库文件可以放在 `storage/app.db`。
- 第一次启动后端时，如果数据库表不存在，需要自动创建表。
- `Document.status` 建议先支持以下状态：
  - `uploaded`：文件已经上传，但还没有解析。
  - `processing`：文件正在处理。
  - `ready`：文件处理完成。
  - `failed`：文件处理失败。
- 上传 PDF 的流程需要变为：
  - 保存 PDF 文件。
  - 创建数据库记录。
  - 返回数据库中的文档 ID。
- `GET /api/documents` 返回结果应包含足够信息，让前端能展示列表。
- 前端可以在首页或文档页显示文档列表。

## 4. 合格标准

- 后端启动后能自动创建 SQLite 数据库文件。
- 上传 PDF 后，数据库中出现一条 `Document` 记录。
- 调用 `GET /api/documents` 能返回上传过的文档列表。
- 重启后端后，再次调用 `GET /api/documents`，之前的文档记录仍然存在。
- 前端刷新后，仍然能显示已上传文档列表。
- 当前任务不要求解析 PDF 页数，也不要求生成页面内容。

## 5. 不合格情况

- 上传记录只保存在内存中，后端重启后丢失。
- 数据库文件没有创建成功。
- 上传 PDF 后没有写入 `Document` 记录。
- `GET /api/documents` 无法返回已上传文档。
- 文档状态字段缺失，导致后续无法表示处理进度。
- 当前任务加入过多 PDF 解析或 LLM 逻辑，使数据库保存功能无法单独验证。
