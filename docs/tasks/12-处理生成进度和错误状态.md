# 任务 12：处理生成进度和错误状态

## 1. 目标

本任务的目标是让用户清楚知道 AI 内容生成到了哪一步，以及某一步失败后如何恢复。

完成后，用户不需要猜测系统是否还在工作，也不会因为某一页生成失败而无法继续使用整份文档。

## 2. 实现内容

- 后端提供文档处理状态接口。
- 前端定期查询生成进度。
- 前端显示课程简介生成状态。
- 前端显示每页讲稿生成状态。
- 页面生成失败时显示错误状态。
- 提供单页重新生成讲稿的能力。
- 前端提供失败页面的重试按钮。

## 3. 实现途径 / 方法 / 手段

- 后端新增 `GET /api/documents/{document_id}/status` 接口。
- 状态接口返回：
  - 文档整体状态。
  - 课程简介是否已生成。
  - 总页数。
  - 已完成讲稿页数。
  - 每页状态。
  - 失败页面的错误信息。
- 前端使用轮询。
  - 轮询是指定时间间隔重复请求接口，例如每 2 秒请求一次。
  - 第一版使用轮询比实时推送更简单。
- 当前端发现文档已经完成处理后，可以停止轮询。
- 后端新增单页重新生成接口，例如 `POST /api/pages/{page_id}/regenerate`。
- 重试时只重新生成当前页讲稿，不重新上传 PDF，也不重新生成全部页面。
- 后端需要记录错误原因。
  - 错误原因可以帮助用户判断是模型配置问题、网络问题还是页面内容问题。
- 前端不能只显示空白，必须显示可理解的状态文案。

## 4. 合格标准

- 上传文档后，前端能显示生成进度。
- 用户能看到课程简介是否已经生成。
- 用户能看到每一页讲稿的状态。
- 某页失败时，该页显示失败信息。
- 某页失败后，用户可以点击重试。
- 重试只影响当前页，不会清空其他页面已生成讲稿。
- 文档全部生成完成后，前端停止无意义的频繁轮询。

## 5. 不合格情况

- 用户无法判断系统是否还在生成。
- 生成失败时页面只显示空白。
- 某一页失败后没有任何恢复方式。
- 重试会重新处理整个文档，导致已成功结果被覆盖。
- 状态接口缺少页级状态，前端只能知道整体成功或失败。
- 前端一直高频轮询，即使文档已经处理完成。
