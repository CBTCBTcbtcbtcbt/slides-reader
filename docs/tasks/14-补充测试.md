# 任务 14：补充测试

## 1. 目标

本任务的目标是为项目核心流程补充测试，确保后续修改不会轻易破坏主要功能。

测试不是为了追求数量，而是为了覆盖最容易出问题、最影响用户使用的路径。

## 2. 实现内容

- 补充后端自动测试。
- 补充前端基础测试。
- 补充一个完整手动验收流程。
- 测试覆盖上传、数据库保存、PDF 解析、页面阅读、讲稿状态、当前页问答等关键功能。
- 记录如何运行测试。

## 3. 实现途径 / 方法 / 手段

- 后端可以使用 `pytest`。
  - `pytest` 是 Python 常用测试框架，可以运行测试函数并报告结果。
- 后端测试建议覆盖：
  - 上传 PDF 成功。
  - 上传非 PDF 被拒绝。
  - 上传后创建 `Document` 记录。
  - PDF 页数解析正确。
  - 每页创建 `Page` 记录。
  - 当前页问答能保存 `ChatMessage`。
- LLM 调用测试中不要真实请求外部模型。
  - 应使用 mock。
  - mock 是测试中的替身对象，用固定结果代替真实外部服务。
  - 这样测试不会依赖网络、API Key 或模型费用。
- 前端可以使用 `Vitest` 和 React 测试工具。
  - `Vitest` 是适合 Vite 项目的 JavaScript 测试框架。
- 前端测试建议覆盖：
  - 上传组件可以渲染。
  - 翻页按钮点击后页码变化。
  - 问答输入框可以输入并提交。
  - 生成状态可以正确显示。
- 手动验收使用一个 3 页 PDF。
- 手动验收流程应完整覆盖：
  - 上传 PDF。
  - 查看课程简介。
  - 等待逐页讲稿生成。
  - 打开每一页查看讲稿。
  - 拖动文字块并刷新验证位置保持。
  - 对某一页提问并刷新验证历史保持。

## 4. 合格标准

- 后端测试可以一条命令运行。
- 前端测试可以一条命令运行。
- 后端测试至少覆盖 PDF 上传校验、数据库保存、PDF 页数解析、问答保存。
- 前端测试至少覆盖上传入口、翻页按钮、问答输入。
- LLM 相关测试不依赖真实 API Key。
- 手动验收流程文档清晰，开发者可以按步骤复现。
- 使用 3 页 PDF 手动验收时，完整流程可以走通。

## 5. 不合格情况

- 没有任何自动测试。
- 测试必须依赖真实 LLM API 才能运行。
- 只能测试单个函数，不能覆盖核心业务流程。
- 没有记录如何运行测试。
- 手动验收步骤不清楚，无法判断项目是否真的可用。
- 关键功能坏了但测试仍然全部通过。
